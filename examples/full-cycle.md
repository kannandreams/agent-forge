# Full-Cycle Example

This example shows how agent-forge skills, workflows, and templates compose
across the full engineering lifecycle: from an idea to a shipped release and
an incident response.

## Scenario

The team needs to add a `GET /api/reports` endpoint to an existing service. A
non-trivial change that spans design, planning, implementation, review, and
release.

## 1. Design — Architecture Design Workflow

The team starts with the architecture design workflow to shape the endpoint
before writing code.

**Skills applied:**
- `artifact-driven-integration` — the endpoint consumes a report artifact
  generated by an internal analytics system
- `service-design` — the endpoint is scoped as a new route in the existing API
  service with a narrow boundary: read-only, no state mutations

**Template used:**
- `architecture_template` — captures the system summary, interface (input:
  query params, output: report list), and operational notes (cached, 5s TTL)

## 2. Plan — Milestone Execution Workflow

With the design in hand, the team breaks the work into an executable plan.

**Skills applied:**
- `project-milestone` — scoped to one milestone: the report endpoint
- `issue-breakdown` — decomposes into 5 tasks: route definition, artifact
  parser, caching layer, tests, API docs

**Template used:**
- `milestone_template` — owner, target date, acceptance criteria per task

## 3. Build — Implementation

The team implements each task. Before committing, they pause at key
integration points to run verification.

**Skills applied:**
- `test-strategy` — maps risk to test layers: unit tests for the parser,
  integration tests for the route + cache, smoke test for the full flow

## 4. Review — Code Review Workflow

Before merging, the change goes through review.

**Skills applied:**
- `code-review` — inspects the diff for correctness, error handling, and test
  coverage
- `test-strategy` — confirms the tests cover the risk points identified in
  step 3

**Workflow used:**
- `code_review_workflow` — orders findings by severity, distinguishes
  blockers from suggestions

## 5. Ship — Release Workflow

The change is ready to ship.

**Skills applied:**
- `release-management` — confirms release scope, validates the build, prepares
  rollback notes

**Workflow used:**
- `release_workflow` — verify → ship → confirm post-release health

**Tool used:**
- `tools/release/changelog.sh` — generates the changelog entry from the merge
  commit

## 6. Respond — Incident Response Workflow

An hour after shipping, the `/api/reports` endpoint returns 500s under load.
The team invokes the incident workflow.

**Skills applied:**
- `incident-response` — confirms scope (only the new endpoint), contains
  impact by toggling a feature flag, records timeline

**Workflow used:**
- `incident_response_workflow` — stabilize → gather facts → mitigate → record
  follow-up

## Reuse Notes

- The design phase chooses the integration boundary before implementation.
- The planning phase converts that boundary into owned work items.
- The review phase checks behavior and test risk against the design.
- The release phase records validation and rollback before shipping.
- The incident phase preserves the timeline and creates follow-up work instead
  of rewriting the release plan.

Reuse this pattern when a change crosses design, delivery, release, and
operations. For smaller changes, select only the relevant workflow.
